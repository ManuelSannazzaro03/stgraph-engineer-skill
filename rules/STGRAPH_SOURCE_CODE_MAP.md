# STGraph Source Code Map

Precise mapping of the STGraph codebase: where STEL is parsed, how expressions are evaluated, how state variables are handled, how built-in functions work, and how `.stg` files are loaded and saved. All line numbers reference the source files in `stgraph-main/`.

## 1. STEL Parsing

### 1.1 Parser Grammar

**File:** `myjep/src/org/nfunk/jep/Parser.jjt` (JavaCC grammar)

The parser is a recursive-descent parser generated by JavaCC/JJTree. It builds an AST of three node types: `ASTConstant` (literals), `ASTVarNode` (variable references), `ASTFunNode` (operators and function calls).

**Production rules** (lowest precedence first):

| Line | Rule | Parses |
|---|---|---|
| 338 | `Start()` | `Expression (SEMI Expression)* EOF` -- semicolons separate sub-expressions |
| 348 | `Expression()` | `AndExpression (OR AndExpression)*` |
| 359 | `AndExpression()` | `EqualExpression (AND EqualExpression)*` |
| 370 | `EqualExpression()` | `RelationalExpression (NE/EQ RelationalExpression)*` |
| 385 | `RelationalExpression()` | `AdditiveExpression (LT/GT/LE/GE AdditiveExpression)*` |
| 408 | `AdditiveExpression()` | `MultiplicativeExpression (PLUS/MINUS MultiplicativeExpression)*` |
| 423 | `MultiplicativeExpression()` | `UnaryExpression (MUL/DIV/MOD/SHARP/SSHARP UnaryExpression)*` |
| 465 | `UnaryExpression()` | Unary `+`/`-`/`!`/`@`, meta-operators, or `PowerExpression()` |
| 619 | `PowerExpression()` | `UnaryExpressionNotPlusMinus [POWER UnaryExpression]` (right-associative) |
| 630 | `UnaryExpressionNotPlusMinus()` | Constants, functions, variables, `(expr)`, `ListExpression`, `DefinedExpression` |
| 654 | `ListExpression()` | `[a,b,c]` (OpList), `[x:y]` and `[x:y:z]` (OpList2), `[]` (empty vector) |
| 673 | `DefinedExpression()` | `{expr}` (OpDefExpr) |
| 681 | `Variable()` | Identifier, optionally followed by `[ListSelector]` -> `get(x, indices)` |
| 723 | `Function()` | Named function calls and function-level meta-operators |
| 559 | `ModifiableOperator()` | Returns the operator token for meta-op resolution |

### 1.2 How Specific Constructs Are Parsed

**Array literals `[a,b,c]`** (line 657): Parsed as `ASTFunNode` with `opSet.getList()`. Children are the element expressions.

**Range literals `[x:y]` and `[x:y:z]`** (lines 662, 667): Parsed as `ASTFunNode` with `opSet.getList2()`. 2 or 3 children.

**Empty vector `[]`** (line 657): `ASTFunNode` with `opSet.getList()` and 0 children.

**Indexing `x[v0,v1]`** (lines 681-720): When a `Variable()` is followed by `[`, the parser calls `ListSelector()` (line 709) which creates an `ASTFunNode` with function name `"get"` (looked up from `jep.funTab`). The variable becomes child 0, each index expression becomes a subsequent child.

**Size operator `@x`** (lines 479-481): Parsed as `ASTFunNode` with `opSet.getSize()`, 1 child.

**Concatenation `x#y`** (line 455-456): `ASTFunNode` with `opSet.getConcatenate()`, 2 children. Same precedence level as `*`/`/`/`%`.

**Decatenation `x##y`** (line 458-459): `ASTFunNode` with `opSet.getDecatenate()`, 2 children.

**Subexpression `{expr}`** (lines 673-678): `ASTFunNode` with `opSet.getDefExpr()`, 1 child.

### 1.3 How Meta-Operators Are Parsed

Meta-operators are parsed at two sites: for operator-level meta-ops (e.g., `+/x`) and for function-level meta-ops (e.g., `max/(x)`).

**Operator-level** (in `UnaryExpression()`, lines 465-556):

The parser calls `ModifiableOperator()` to capture the left-hand operator, then checks for `/` (reduction), `\` (scan), or `|` (pair-scan) tokens:

| Lines | Syntax | Result |
|---|---|---|
| 525-533 | `op / expr` | `ASTFunNode(op, 2 children: ASTConstant("/"), expr)` |
| 545-553 | `op \ expr` | `ASTFunNode(op, 2 children: ASTConstant("\\"), expr)` |
| 535-543 | `op \| expr` | `ASTFunNode(op, 2 children: ASTConstant("\|"), expr)` |
| 483-495 | `op / [n] expr` | `ASTFunNode(op, 3 children: ASTConstant("/"), ASTConstant(n), expr)` |
| 511-523 | `op \ [n] expr` | same with `"\\"` |
| 497-509 | `op \| [n] expr` | same with `"\|"` |

**Function-level** (in `Function()`, lines 723-836):

Named functions followed by `/`, `\`, or `|` before the argument list:

| Lines | Syntax | Result |
|---|---|---|
| 789-803 | `func / (args)` | `ASTFunNode(func, children: ASTConstant("/"), args...)` |
| 821-835 | `func \ (args)` | same with `"\\"` |
| 805-819 | `func \| (args)` | same with `"\|"` |
| 744-787 | Dimensional variants `func/[n](args)` | 3-child pattern with dimension constant |

**Key insight:** The parser encodes the meta-operator type as a string constant (`"/"`, `"\\"`, `"|"`) pushed as the first child of the function node. At evaluation time, `STDyadicFun.run()` detects this string to dispatch the meta-operation.

### 1.4 Function vs Variable Disambiguation

**Lines 634-641:** The parser uses a semantic LOOKAHEAD: if the identifier exists in `jep.funTab` OR (exists in `jep.symTab` AND starts with `_` but not `__`), it is parsed as a `Function()`. Otherwise as a `Variable()`. The `_` prefix convention is what enables user-defined callable functions.

## 2. Expression Evaluation

### 2.1 The Evaluator

**File:** `myjep/src/org/nfunk/jep/EvaluatorVisitor.java`

Uses the visitor pattern on the AST. Entry point: `getValue(Node topNode, SymbolTable symTab)` at line 84.

1. Clears the evaluation stack (line 96)
2. Calls `topNode.jjtAccept(this, null)` (line 101) -- dispatches to `visit()` per node type
3. Asserts exactly 1 item on stack (line 112)
4. Pops and returns the result (line 117)

### 2.2 AST Node Type Evaluation

**`ASTConstant`** (lines 292-295): Pushes the constant value onto the stack.

**`ASTVarNode`** (lines 259-286):
1. Gets `Variable var = node.getVar()` (line 270) -- the `Variable` object was bound at parse time
2. If `var == null`: throws `ParseException` (line 271-273)
3. Gets `Object temp = var.getValue()` (line 276) -- reads current value from the symbol table
4. If `trapNullValues` is true (default) and `temp == null`: throws `ParseException` with "variable not found" (line 278-281)
5. Pushes value onto stack (line 284)

**`ASTFunNode`** (lines 212-253): Three dispatch paths:

| Path | Interface | Behavior |
|---|---|---|
| `SpecialEvaluationI` (line 228) | Function controls its own child evaluation | Used by assignment operators |
| `CallbackEvaluationI` (line 232) | Function receives the AST node and evaluator | Used by `if`, `iter`, `array`, `function` -- lazy evaluation |
| Standard (line 240) | All children evaluated first, then `run(stack)` | Used by all arithmetic ops and most functions |

For the standard path: `node.childrenAccept(this, data)` recursively evaluates all children left-to-right, each pushing one value onto the stack. Then `pfmc.run(stack)` pops arguments and pushes the result.

**There is no distinction between operators and functions at the evaluation level.** Both are `ASTFunNode` with a `PostfixMathCommandI` attached. The operator's PFMC is set via `Operator.setPFMC()` in `STInterpreter`'s constructor.

### 2.3 Variable Resolution

Variables are resolved by reference. At parse time (`Parser.jjt` line 696), `jjtThis.setVar(symTab.getVar(identString))` binds the `Variable` object to the AST node. During simulation, when `setNodeValue()` calls `interpreter.addVariable(nodeName, value)`, it updates the `Variable` object's value in the symbol table. Subsequent evaluations of any expression referencing that variable automatically see the new value -- no re-binding needed.

If a variable name doesn't exist in the symbol table at parse time, `symTab.getVar()` returns null, and the `Variable` object is null. At evaluation time, this triggers the `var == null` check at line 271.

### 2.4 Pre-Parsing (Before Simulation Starts)

**File:** `stgraph/src/it/liuc/stgraph/STGraphExec.java`, method `preParseExpressions()` at line 1285.

Called from `preCompute()` at line 1099. Converts all expression strings into AST trees before the simulation loop begins, so parsing is not repeated at each step.

**For algebraic nodes** (lines 1287-1301): Calls `interpreter.preParseExpression(node.getExpression())`, stores result in `node.topOfTreeValue`. If the node has a saturating expression (submodel input), uses the super-graph's interpreter instead.

**For state nodes** (lines 1302-1307): Pre-parses `stateInit` into `node.topOfTreeState`. If state-with-output, also pre-parses `expression` into `node.topOfTreeValue`. For plain state nodes, `topOfTreeValue = topOfTreeState`.

**`STInterpreter.preParseExpression()`** (line 430): Splits on `;` (semicolons), parses each sub-expression separately, returns `ArrayList<Node>`. Multi-statement expressions return multiple AST roots.

### 2.5 Runtime Expression Evaluation

**`STInterpreter.evalExpression(STNode node, ArrayList<Node> topOfTree)`** at line 475:

1. Calls `resetPrivateVars()` (line 476) -- resets `$0`, `$1`, `$a0`..`$a3`, `$i`, `$v0`..`$v3`, `$w0`..`$w3`, `$p0`..`$p2`, `$numArgs` to empty Tensor; resets `numDefExpr = 0`
2. Creates fresh stacks for `cnop` and `nde` (lines 479-480) -- re-entrant call tracking
3. For multi-statement expressions: evaluates all but the last sub-expression, storing results in `$v0`, `$v1`, etc. (lines 482-485)
4. Evaluates the last sub-expression as the return value (line 486-487)
5. On success: clears `node.evalErrDescription`, returns result
6. On error: sets `lastErrorNode`, builds error description, sets `node.evalErrDescription`, returns null

## 3. Tensor Polymorphism Engine

### 3.1 STDyadicFun -- Dyadic Polymorphism

**File:** `stgraph/src/it/liuc/stgraph/fun/STDyadicFun.java`

The `run()` method at line 64 is the central polymorphism dispatcher for ALL dyadic operators. Subclasses only implement `abstract Object exec(Tensor x1, Tensor x2)` which operates on scalar tensors.

**2-parameter standard dispatch** (lines 70-176):

| Condition | Behavior | Lines |
|---|---|---|
| Either operand is empty (size == -1) | Return the empty operand | 76-84 |
| Both scalars (size == 0) | Call `exec(scalar1, scalar2)` | 87-88 |
| Left scalar, right non-scalar | Broadcast: `exec(scalar, each_element)` | 91-94 |
| Right scalar, left non-scalar | Broadcast: `exec(each_element, scalar)` | 98-101 |
| Same dimensions | Element-wise: `exec(elem_i, elem_i)` | 124-128 |
| Right is (n-1)-order, matching first dims | Broadcast right along last dim | 132-152 |
| Left is (n-1)-order, matching first dims | Broadcast left along last dim | 153-173 |

**Meta-operator dispatch** (lines 178-327, when first parameter is a String):

The string `"/"`, `"\\"`, or `"|"` was pushed as a constant child by the parser. `STDyadicFun.run()` detects `param1 instanceof String` and dispatches:

| Selector | Lines | Algorithm |
|---|---|---|
| `"/"` (reduction) | 209-220 | For each sub-tensor along last dim: fold left with `exec()` |
| `"\|"` (pair-scan) | 222-233 | For each pair of adjacent elements: `exec(elem[j], elem[j+1])` |
| `"\\"` (scan) | 235-244 | Cumulative fold: `result[0]=elem[0]; result[i]=exec(result[i-1], elem[i])` |
| Dimensional variants | 247-327 | Same three operations but along dimension `dim`, using `getGivenDimTensor()` |

**Example trace of `+/[1,2,3,4]`:**
1. Parser creates `ASTFunNode(OpAddition)` with children: `ASTConstant("/")`, `ASTFunNode(OpList, [1,2,3,4])`
2. Evaluator evaluates children: pushes `"/"`, pushes `Tensor([1,2,3,4])`
3. `OpAddition.run(stack)` -> `STDyadicFun.run(stack)` pops both
4. Detects `param1 instanceof String`, selector = `"/"`
5. Gets last-dim tensor: `[1,2,3,4]`
6. Fold: `exec(1,2)=3`, `exec(3,3)=6`, `exec(6,4)=10`
7. Pushes `Tensor(10.0)`

### 3.2 STMonadicFun -- Monadic Polymorphism

**File:** `stgraph/src/it/liuc/stgraph/fun/STMonadicFun.java`, `run()` at line 47.

Simpler than dyadic. Subclasses implement `abstract Object exec(Tensor d)` on scalar tensors.

|---|---|---|

| `STATE_PRE_EVALUATING` | 1 | Initial evaluation pass (errors suppressed) |
| `STATE_STEPPING` | 2 | Step-by-step execution |
| `STATE_TIMING` | 3 | Timer-based execution |

| `STATE_BATCHRUNNING` | 5 | Batch execution |

### 4.2 Step 0 -- State Initialization

In `computeStateValues()` (line 1407), when `firstCall == true`:

1. Evaluates `topOfTreeValue` (which equals `topOfTreeState` = pre-parsed `stateInit`) via `interpreter.evalExpression(node, node.topOfTreeValue)`
2. Result becomes the node's initial value
3. `this` is NOT set before this evaluation -- `stateInit` must not reference `this`

In `computeNextStates()` (line 1487), when `firstCall == true && integrationPhase == 1`:

1. Re-parses `stateTransition` into `topOfTreeState` (overwriting the stateInit parse tree)
2. Sets `this` = current state via `interpreter.setVarValue("this", node.getCurrentState())`
3. Evaluates `topOfTreeState` to compute the first transition value
4. Stores result in `node.nextState`

### 4.3 Step >0 -- State Advancement

In `computeStateValues()`, when `firstCall == false`:

1. Simply reads `node.getNextState()` -- the value computed in the previous step's `computeNextStates()`
2. Sets it as the current node value

In `computeNextStates()`, when `firstCall == false`:

1. Sets `this` = current state
2. Evaluates `topOfTreeState` (now the transition expression)
3. Stores result in `node.nextState`

### 4.4 State-With-Output Nodes

State-with-output nodes (valueType=2) are handled differently from plain state nodes:

- They appear in `sortedNodeList1` or `sortedNodeList2` (not just in `stateNodeGlobalList`)
- Their `expression` is evaluated in `computeInputs()` or `computeSync()` (depending on `fromState` flag)
- In `computeInputs()`/`computeSync()`: `currentState` is set from `stateInit` (firstCall) or `nextState` (subsequent), then `this` is set to `currentState`, then `me` is set to the transition result, then the `expression` is evaluated as the output
- The output value (from `expression`) is what downstream nodes see
- The state (from `stateInit`/`stateTrans`) is internal

### 4.5 The `integral()` Function

**File:** `stgraph/src/it/liuc/stgraph/fun/Integral.java`, `run()` at line 52.

**Validation:** Line 59 checks `node.isState()` -- error if used outside a state node.

**1 parameter** (line 66): `param1 = argument`, `param2 = graph.thethis` (the current `this` value).

**Euler** (lines 92-101):
```
result[i] = param2[i] + param1[i] * timeD    // i.e., this + x * timeD
```

**RK2** (lines 109-145, 3 integration phases):
- Phase 1: Euler step `this + x * timeD`
- Phase 2: Stores argument in a static buffer (used by phase 3)
- Phase 3: `this + x*timeD/3 + buffer*2*timeD/3`

**RK23** (lines 168-219, 5 integration phases):
- Phase 1: Euler `this + x * timeD`
- Phase 2: buffer[0] stores argument
- Phase 3: Euler again
- Phase 4: buffer[1] stores argument
- Phase 5: `this + x*2*timeD/9 + k1*timeD/3 + k2*4*timeD/9`

**Non-obvious behavior:** The `index` field at line 41 is `static` -- shared across ALL `Integral` instances. This tracks which buffer slot to use during RK phases. A known edge case (TODO comment at line 129/200): if `integral()` is inside an `if()` branch that evaluates differently across RK phases, the buffer index gets out of sync.

### 4.6 Save/Restore for RK Integration

In `compute()` (line 1113):

- `saveComputationalState()`: Copies all state node current values into `stateBuffer`, `stateBuffer2`, `stateBuffer3` (triple-buffered for RK23)
- `saveTime()` / `restoreTime()`: Saves/restores `currStep`, `currTime`, interpreter time variables
- `setTempTimeD(factor)`: Temporarily modifies `timeD` for the fractional step
- `restoreComputationalState()`: Copies saved values back to state nodes

### 5.1 Function Base Classes

| Base Class | File | Pattern | Used By |
|---|---|---|---|
| `STMonadicFun` | `fun/STMonadicFun.java` | Element-wise map over scalar elements | `Cos`, `Sin`, `Exp`, `Log`, `Sqrt`, `Sign`, `Int`, etc. |

### 5.2 Key Function Implementations

**`If.java`** (`fun/If.java`, `evaluate()` at line 47) -- Implements `CallbackEvaluationI`:

- Variable args, requires odd count >= 3 (line 50)
- **Scalar conditions** (line 75-76): Short-circuits -- evaluates conditions left-to-right, returns the first matching value. Falls through to default.
- **Tensor conditions** (lines 77-104): Per-element first-match semantics. Creates result tensor and tracker tensor `x`. For each condition-value pair: for each element position `j`, if `x[j] != 1` (unassigned) AND condition `c[j]` is true, assigns `result[j]` from value and marks `x[j] = 1`. After all pairs, fills unassigned positions from default. This means different elements of a vector can match different conditions.

**`Iter.java`** (`fun/Iter.java`, `evaluate()` at line 56) -- `iter(data, expr, zero)`:

- Saves/restores `$0`, `$1`, `$i` for reentrancy (lines 160-173)
- **Scalar data** (lines 73-94): Interprets as a range 0..N-1. `$0`=accumulator, `$1`=`indexOrigin+i`, `$i`=same
- **Vector data** (lines 102-123): `$1` = each element, `$0` = running result
- **Higher-order data** (lines 125-156): Iterates along the **last dimension**, `$1` gets each sub-tensor slice
- Empty vector input (size <= 0): returns `zero` immediately (lines 97-99)

**`CreateTensor.java`** (`fun/CreateTensor.java`, `evaluate()` at line 45) -- `array(dims, expr)`:

- First arg: scalar or vector of dimension sizes (line 51-54)
- Any dimension == 0 -> returns empty tensor (line 61)
- Sets `$i0`..`$i(n-1)` to current multi-dim index for each element (lines 68-73)
- Sets `$p0`, `$p1`, `$p2` to previous 1, 2, 3 results -- enables recurrence relations (line 72)
- **Only scalar results per element** -- if the expression returns a tensor of size > 1, `0.0` is stored instead (lines 76-86)

**`CreateFunction.java`** (`fun/CreateFunction.java`, `evaluate()` at line 43):

- Returns the **raw AST node** (`node.jjtGetChild(0)`) without evaluating it
- This is how `function(expr)` works: the node stores a parse tree reference, not a value
- When another node calls `_funcName(args)`, `CustomFun2.run()` evaluates this stored AST

**`CustomFun2.java`** (`fun/CustomFun2.java`, `run()` at line 67):

Full re-entrant execution engine for custom functions:

1. Sets `$numArgs` (line 70)
2. Pops all arguments from stack in reverse order (lines 71-72)
3. Pushes `$numArgs` and `numDefExpr` onto static stacks for reentrancy (lines 73-75)
4. **Saves the entire JEP evaluation stack** into `stackImage` (lines 76-77)
5. Saves current `$a0`..`$a3`, sets them to actual arguments (lines 78-80)
6. Evaluates semicolon-separated sub-expressions, storing in `$v0`, `$v1`, etc. (lines 81-84)
7. Evaluates last sub-expression as return value (lines 85-86)
8. Restores stack, `$a` vars, `cnop`/`nde` stacks, pushes result (lines 95-101)
9. If result is a `String`, treated as a localizable error message (line 91-93)

**`Get.java`** (`fun/Get.java`, `run()` at line 49) -- `get(data, i0, i1, ...)`:

- Requires >= 2 params. Data must be non-scalar. Indexes must be scalars or vectors.
- Adjusts indexes by `indexOrigin` (line 91)
- Uses `getSubTensorByMajorDim()` and `transpose()` for multi-dim slicing
- Calls `dimensionMinimize()` at the end to reduce unnecessary dimensions

**`Set.java`** (`fun/Set.java`, `run()` at line 48) -- `set(data, i0, ..., value)`:

- Scalars are auto-promoted to 1-element vectors (lines 67-73) -- non-obvious
- Builds index mapping tensor, then writes values at mapped positions

**`Integral.java`** -- see section 4.5 above.

**`Eval.java`** (`fun/Eval.java`, `run()` at line 45) -- `eval(string)`:

- Argument must be a `String` (line 48)
- Dynamically parses and evaluates the string as STEL at runtime (lines 52-54)
- Errors are wrapped with `STTools.replaceCRwithBR`

**`ReadFromXLS.java`** (`fun/ReadFromXLS.java`, `run()` at line 46):

- **4 params** (lines 48-68): `readFromXLS(file, sheet, row, col)` -> single scalar. Indices are 1-based.
- **6 params** (lines 70-118): `readFromXLS(file, sheet, r1, c1, r2, c2)` -> range as vector or matrix.
- Non-numeric cells silently return `0.0` (line 133)
- Workbooks are cached via `handleResource()` (line 59)

### 5.3 Distribution Function Calling Convention

**File:** `stgraph/src/it/liuc/stgraph/fun/STDistributionFun.java`, `run()` at line 45.

| Args | Syntax | Behavior |
|---|---|---|
| 0 | `f()` | Random sample with default params |
| 1 | `f(v)` | Random sample with specified param vector `v` |
| 2 | `f(x, sel)` | `sel=0`: PDF, `sel=1`: CDF, `sel=2`: inverse CDF; default params |
| 3 | `f(v, x, sel)` | Same with specified params |

When `x` is a tensor, applies element-wise. For inverse CDF (`sel=2`), validates each value is in `[0,1]`.

### 5.4 Interpolation Function Pattern

**File:** `stgraph/src/it/liuc/stgraph/fun/STInterpolationFun.java`, `run()` at line 46.

Always 3 params: `f(nx, ny, xx)`. Both `nx` and `ny` must be vectors. Calls `init(nx, ny)` to build the model, then applies `exec(x)` element-wise over `xx`. Result tensor has the same shape as `xx`.

## 7. Non-Obvious Behaviors

### 7.1 Meta-Operators Are Not Separate Classes

Meta-operators (`f/x`, `f\x`, `f|x`) are handled entirely within `STDyadicFun.run()`. The parser encodes the meta-op type as a string constant (`"/"`, `"\\"`, `"|"`) pushed as the first child of the operator's `ASTFunNode`. At evaluation, `STDyadicFun.run()` detects `param1 instanceof String` and dispatches accordingly using the same `exec()` kernel.

### 7.2 `function()` Returns an AST Node, Not a Value

`CreateFunction.evaluate()` returns the raw AST child node without evaluating it. This makes the node's value a parse tree reference. When `CustomFun2.run()` encounters a call to `_funcName(args)`, it evaluates this stored AST with `$a0`..`$a3` bound to the arguments.

### 7.3 Custom Function Reentrancy via Stack Saving

`CustomFun2.run()` saves and restores the entire JEP evaluation stack (line 76-77), all `$a` variables, and the `cnop`/`nde` static stacks. This enables recursive function calls and nested custom function invocations.

### 7.4 `Integral.java`'s Static `index` Field

The `index` field at line 41 is `static` -- shared across ALL `Integral` instances. It tracks which buffer slot to use during RK multi-phase evaluation. This means all `integral()` calls in a model share the same phase counter. A known bug (TODO comments at lines 129 and 200): if `integral()` appears inside an `if()` branch that evaluates differently across RK phases, the buffer index can get out of sync, producing incorrect results.

### 7.5 Pre-Parsed Trees Are Mutated During Execution

In `computeNextStates()`, when `firstCall == true && integrationPhase == 1`, the code re-parses `stateTransition` into `topOfTreeState`, **overwriting** the `stateInit` parse tree that was stored there during `preParseExpressions()`. After step 0, `topOfTreeState` permanently holds the transition expression.

### 7.6 `STGraphImpl.load()` Reads Children Sequentially

The parser uses `theEnum.nextElement()` without checking element names. If `.stg` XML elements are reordered, the parser will cast the wrong element to the wrong handler, causing ClassCastExceptions or corrupt data.

### 7.7 Nodes Are Alphabetically Sorted on Save

`save()` sorts nodes by name before writing (line 783). This means node order in the XML file changes on every save if nodes are added. The order is irrelevant for loading (nodes are stored in a map by name), but it can cause unnecessary diffs in version control.

### 7.8 `If.java` Per-Element First-Match for Tensors

When conditions are tensors (not scalars), `If.java` applies first-match semantics **per element position**. Each element independently picks the first true condition's corresponding value. This is not short-circuit evaluation -- all conditions and values may be evaluated even if earlier conditions matched for all elements.

### 7.9 `ReadFromXLS` Silently Returns 0.0 for Non-Numeric Cells

At line 133, any exception during cell reading (non-numeric cell, out of bounds, format error) is caught and `0.0` is returned. There is no error signal -- the model silently uses zero.

### 7.10 `Get.java` Dimension Minimization

After slicing, `Get.java` calls `dimensionMinimize()` which strips unnecessary outer dimensions. For example, accessing `m[0]` on a matrix returns a vector (not a 1-row matrix). This can cause surprising dimension changes when chaining operations.

### 7.11 `Set.java` Auto-Promotes Scalars

At lines 67-73, `Set.java` auto-promotes scalar data to a 1-element vector before processing. This means `set(5, 0, 10)` works despite `5` being a scalar, returning `[10]`. The caller may not expect the type change.

### 7.12 Empty Vector Polymorphism Fallback

In `STDyadicFun.run()`, lines 76-84: if either operand is an empty non-scalar Tensor (`getSize() == -1`), the result is that empty operand itself. This means `[]+5` returns `[]`, and `+/[]` returns `Tensor(0.0)` (the fold start value). But some functions don't handle empty input, leading to `NullPointerException` on `value` access since empty tensors have `value=null`.
